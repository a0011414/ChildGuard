# ChildGuard プロジェクト 引き継ぎ資料

このフォルダ（ChildGuard）を**ワークスペースとして開き、続きを進める**際の引き継ぎ用ドキュメントです。

---

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| **フォルダ名** | ChildGuard（目指すところ＝子どもを守る、で統一） |
| **アプリ名** | 未定。別の視点で選ぶ想定。候補は `アプリ名候補.md` 参照。 |
| **目的** | 親子で決めたルールに従い、中毒性の高いサービスから子どもを守る。既存のペアレンタルコントロールを超え、**MCP（エージェント）**で「ルールに従って」制御する。 |

---

## 2. 背景・きっかけ

- **詳細**: `背景・きっかけ.md` を参照。
- **要約**: 高校1年の息子が入学から iPhone を使い始め、最近タガが外れて依存症気味になり、取り上げざるを得なくなった経験がきっかけ。目指すのは「取り上げる前」に、親子のルールと仕組みでブレーキをかけられるようにすること。

---

## 3. コンセプト・機能イメージ（これまでの議論）

- **親子でルールを決める** → そのルールに「従って」動く仕組み。
- **例**: 「〇時間超えそうなので止めた」と親に通知し、自動で使用制限。
- **例**: アプリのインストール・写真アップロードなども、ルール・ガイドラインを参照して判断。
- **技術的イメージ**: MCP（エージェント）がルールを参照し、端末やサービスを制御する。

---

## 4. スコープ・スタンス（重要）

- **現時点の必要十分**: **自分の家だけで使えればいい**。**金儲けは必要ない**。
- **一方で**: 「世界の親子を救えるアプリになるかもしれない」と考えており、**必要なら会社も作ろうと思っている**。
- 引き継ぎ先では、**「会社を作るかどうか」の判断**について、フローチャート等で整理してほしい、という要望あり（下記 5. に記載）。

---

## 5. 「会社を作るか」の判断について

「自宅だけで使えればいい・金儲け不要」だが「世界の親子を救えるかも→必要なら会社を作る」というスタンスのため、**いつ・なぜ会社を検討するか**を判断できるようにしておく。

### 5.2 判断のポイント（引き継ぎ用メモ）

| 状況 | 会社を作る必要性 |
|------|------------------|
| 自宅だけで使う | **不要**。個人開発・個人利用のまま。 |
| 無料で多くの人に配布（OSS・ストア等） | **必須ではない**。免責・利用規約・README で注意喚起すれば個人でも可能（国・法域により要確認）。 |
| 有料で販売・継続サポート・問い合わせ対応 | **検討した方がよい**。収益・契約・責任の受け皿として法人があると安心。 |
| 商標・ブランドを守りたい・契約を法人で結びたい | **検討した方がよい**。責任の所在を明確にできる。 |
| チームで開発・採用したい | **検討した方がよい**。雇用・対価・IP の扱いを整理しやすい。 |

**結論のメモ**: まずは**自宅用で完成させる**ことを最優先にし、「届けたい」が具体化した段階で、上記フローと表を参照して会社の要否を判断すればよい。

## 7. 今後の進め方・次のステップ（提案）

1. **要件の整理**: 自宅で「誰が・何を・どのルールで」制御したいか、箇条書きでよいのでまとめる。
2. **技術選定**: MCP/エージェントをどこに載せるか（例: 自宅サーバ、端末内、クラウド）、どのOS（iOS/Android）から手を付けるか。
3. **プロトタイプ**: 最小単位（例: 1 つのルール＋1 つの制御）で動くものを作り、自宅で試す。
4. **会社の要否**: 自宅用が形になってから、「届けたい」が具体化した時点で、上記 5. のフローで判断。
